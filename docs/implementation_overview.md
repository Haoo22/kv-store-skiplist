# KV-Store 实现综述

本文档从实现角度概述项目的关键设计和模块分工。

## 1. 目标

KV-Store 提供一个结构清晰、可验证、可扩展的轻量级键值存储实现，覆盖：

- 网络接入
- 协议解析
- 内存索引
- 持久化恢复
- 测试与 benchmark

项目的主要使用场景是读多写少的有序键值访问：读请求直接命中内存索引，写请求通过 WAL 保证重启后的可恢复性。相比只实现一个内存容器，本项目把网络接入、协议解析、存储索引、持久化恢复和验证脚本串成一条完整链路。

## 2. 主要组件

### 2.1 服务端

服务端采用单线程 Reactor 模型，基于 `epoll` 和非阻塞 socket 处理连接、读写和命令执行。

网络线程负责完整请求路径，包括读取字节流、提取命令、执行存储操作和写回响应。这个设计避免了跨线程调度带来的额外复杂度，适合作为轻量级 KV 服务的主循环。

### 2.2 协议层

协议层使用按行文本协议，`LineCodec` 负责按 `\r\n` 提取完整命令，`CommandProcessor` 负责解析和调度。

文本协议便于手工调试，也便于其他语言编写客户端。只要客户端通过 TCP 发送以 `\r\n` 结尾的命令，就可以访问服务端提供的 KV 操作。

### 2.3 存储层

存储层由 `KVStore` 和 `SkipList` 组成：

- `KVStore` 提供统一接口
- `SkipList` 维护有序键空间
- 单 key 更新路径使用节点级锁
- 范围查询依赖有序底层链表

`KVStore` 屏蔽 WAL 与内存索引的组合细节；调用者只需要使用 `Put`、`Get`、`Delete` 和 `Scan`。跳表维护有序键空间，因此除了点查询，也能支持按 key 范围扫描。

### 2.4 WAL

WAL 负责写前日志记录和重启恢复：

- 写操作先追加日志
- 服务启动时重放日志
- 尾部损坏记录会被跳过

WAL 只记录会改变状态的操作，例如 `PUT` 和 `DEL`。`GET` 和 `SCAN` 不写日志，因此读多写少场景下主要开销仍集中在内存索引读取路径。

## 3. 支持的能力

- 点查询
- 范围查询
- 删除
- 基础持久化
- 重启恢复
- 网络吞吐测试
- 进程内基线对比

## 4. 基线实现

项目提供 `std::map + mutex` 版本作为进程内对比基线，用于观察跳表实现与 WAL 的相对性能开销。
