# KV-Store 论文驱动重构执行计划

本计划是“按论文重构项目”的执行版清单。它不是方向概述，而是后续逐阶段实施、验收和更新 README 的依据。

## 1. 固定边界

- 主线服务端固定为单线程 Reactor。
- 线程池方案仅保留为实验反例与问题链路，不重新合回主线。
- 优先做“论文与项目一致化”，再做“工程整理”，最后做“展示增强”。
- 每个阶段结束后都要同步更新 README。

## 2. 阶段划分

### 阶段 1：论文口径冻结

目标：
- 建立论文目标与项目现状的映射
- 冻结后续所有表述边界

交付物：
- `docs/internal/thesis_alignment.md`
- README 中的论文入口与边界说明

验收标准：
- 不再出现“当前主线是多线程 worker server”这类错误表述
- benchmark、协议验证、demo 三条线清晰分离

### 阶段 2：项目结构与文档重构

目标：
- 让仓库结构更服务论文阅读、答辩讲解和实验复现

计划动作：
- 梳理 `docs/` 下的论文支撑材料
- 明确模块职责、实验分类、答辩展示入口
- 给 README 增加“先看哪里”的导航

交付物：
- 模块总览文档
- 实验分类文档
- Demo 使用与定位文档

验收标准：
- 新读者不看开发日志，也能理解当前仓库结构和实验边界

### 阶段 3：核心代码边界整理

目标：
- 让主线实现更适合论文描述和答辩复述

计划动作：
- 审查服务端、协议层、WAL、benchmark 的接口与帮助信息
- 统一错误输出、协议响应、参数说明
- 清理与主线叙事冲突的遗留表述

交付物：
- 更稳定清晰的主线入口
- 与 README 对齐的命令行帮助和输出格式

验收标准：
- 文档中的命令、代码中的行为、实际输出三者一致

### 阶段 4：验证链路重构

目标：
- 把“代码能跑”变成“实验可复现、论文可引用”

计划动作：
- 按层次整理单元测试、协议验证、正式 benchmark、WAL 恢复验证
- 统一实验命令、输出解释和环境限制说明
- 必要时补轻量验证脚本

交付物：
- 清晰的验证入口和实验说明
- 可直接写入论文实验部分的步骤模板

验收标准：
- 每条验证路径都能明确回答“用途是什么、怎么跑、输出怎么看”

### 阶段 5：答辩展示链路整理

目标：
- 让 demo 真正服务答辩，而不是和正式 benchmark 混淆

计划动作：
- 审查 demo 文案、指标标签、模式说明
- 明确“真实事件映射”与“非正式 benchmark”边界
- 整理答辩演示流程

交付物：
- Demo 说明文档
- 答辩演示步骤文档

验收标准：
- Demo 页面与文档不再误导老师把演示结果当成正式实验数据

### 阶段 6：论文支撑材料沉淀

目标：
- 把仓库成果沉淀成论文正文、答辩 PPT 可直接复用的材料

计划动作：
- 提炼系统架构说明、模块设计说明、实验方法说明、问题链路总结
- 补架构图、数据流描述、实验摘要

交付物：
- 论文可复用材料包

验收标准：
- 论文正文和答辩 PPT 能直接引用这些材料，而不是临时从 README 长文里抽取

## 3. 优先级

- `P0`：论文口径冻结、README 与事实一致、实验边界清晰
- `P1`：核心代码边界整理、验证链路重构
- `P2`：demo 讲述力增强、论文附图和附录材料沉淀

## 4. 执行约束

- 每阶段开始前先检查当前仓库状态。
- 发现实现与 README 不一致时，优先指出并修正。
- 任何代码修改完成后，都必须同步更新 README。
- 如果因环境限制无法验证，必须把卡点写入 README 或对应文档。
